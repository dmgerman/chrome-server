=browsel= is a module that allows Emacs and a Web browser to
communicate with each other, in both directions.  Right-click on a
page in the Web browser (Chrome or Firefox) and have it sent to Emacs (org-capture, save-page,
archive a ChatGPT conversation, save a YouTube video for later).
Going the other way: from Emacs query browser state (what tabs are
open?  what's the page title?  what's the video's current time?) and
act on it.

=browsel= is composed of two parts:

1. *An Emacs side*.  A WebSocket server on =127.0.0.1:9130= that
   registers handlers for actions initiated from the browser, plus
   an elisp API for actions Emacs initiates.
2. *A browser extension*.  A WebExtension that connects to the Emacs
   server, registers the right-click menu items, implements the
   toolbar icon and the popup, and routes requests in both
   directions.  Two builds ship from the same sources: a Chrome
   Manifest V3 build and a Firefox Manifest V2 build.  Both speak
   the same protocol and identify themselves to Emacs by name
   (=chrome=, =firefox=) so a single Emacs can drive both at once.

Emacs and the extension must be running.  The extension auto-reconnects, so restarting Emacs is non-disruptive.  Setup for each side is below.

Security has been a major concern in the design of this module. Please see [[*Security][Security]] below
for more information.

=browsel= was inspired by [[https://github.com/bitspook/spookfox][spookfox]], a similar Emacs/Firefox
integration.  The protocol and handler-registry pattern are adapted
from it.  A major enhancement is that it provides
two-way communication: Emacs can invoke browser handlers, and the
browser can invoke Emacs handlers.

For implementation details — adding new actions, the wire protocol,
the JSON schemas — see =CONTRIBUTING.org=.

* Quick setup

The full setup is documented in the *Setup* section below; this is
the short form.

1. *Configure and start the Emacs side.*  Add a =use-package= form to
   your Emacs configuration that loads =browsel=, requires the
   optional backends for the features you want, and starts the
   WebSocket server:

   #+begin_src emacs-lisp
   (use-package browsel
     :load-path "/path/to/browsel"
     :config
     (require 'browsel-www)         ; SAVE_PAGE
     (require 'browsel-chatgpt)     ; CHATGPT
     (require 'browsel-youtube)     ; YOUTUBE + YOUTUBE_TRANSCRIPT
     (require 'browsel-babel)       ; org-babel browsel-js blocks
     (require 'browsel-tab-manager) ; M-x browsel-tab-manager
     (require 'browsel-url-handler) ; browsel-browse-url + browsel-url-routes
     (browsel-start))
   #+end_src

   Comment out any =require= for a feature you do not want.

2. *Load the browser extension.*  Both builds are committed already
   built; no build step is required.

   In Chrome:
   1. Open =chrome://extensions= and turn on *Developer mode* (top right).
   2. Click *Load unpacked* and select =extension/build/chrome/=.
   3. (Only if you want JavaScript evaluation) open the extension's
      *Details* page and toggle *Allow User Scripts* on.

   In Firefox:
   1. Open =about:debugging= and click *This Firefox*.
   2. Click *Load Temporary Add-on…* and select
      =extension/build/firefox/manifest.json=.

The popup should show ●  *Connected* within a few seconds.  If it
does not, click *Reconnect*.

* Features

** In the Web browser

- Right-click menu items that trigger Emacs actions: org-capture,
  org-roam-capture, open-in-eww, save-page, save-chatgpt,
  add-youtube-video, save-youtube-transcript. The actions
  can inject/execute javascript to build the request to emacs.
- Configurable keyboard shortcuts for the same actions.
- A toolbar popup with live connection status, per-tab permission
  controls, one-click buttons for every action, and a list of tabs
  with granted JS-eval permission (click an entry to switch to that
  tab).
- Per-tab consent before executing arbitrary JavaScript evaluation: an in-page overlay prompts for permission, and the
  toolbar icon turns red while consent remains in effect.

** In Emacs

- =M-x browsel-tab-manager= — select open tab
  (ordered in most-recently-used first).
- =M-x browsel-org-link= — insert an Org link to the active
  browser tab at point.
- =M-x browsel-selection= — insert the current text selection
  from the active browser tab at point.
- =M-x browsel-scroll-page= — scroll the active tab one viewport
  down; numeric prefix scrolls N pages (negative = back); bare =C-u=
  scrolls one page back.
- =M-x browsel-video-toggle-play= — toggle play/pause on the first
  =<video>= element in the active tab.
- =M-x browsel-video-advance= — seek the first =<video>= forward by
  5 seconds; numeric prefix sets a different N (negative = back);
  bare =C-u= seeks 5 seconds back.
- =M-x browsel-video-current-time= — insert the current playback
  time of the first =<video>= as =H:MM:SS= / =M:SS= / =S= (leading
  zero components dropped).
- =M-x browsel-youtube-org-link= — insert an Org link to the
  active YouTube tab at its current playback time (=&t=Ns= in the
  URL, =title @ timestamp= as the description); errors if the tab
  is not on youtube.com / youtu.be or has no video element.
- =M-x browsel-set-default-client= — when both browsers are
  connected, pick which one interactive commands address; =C-u=
  clears the setting.
- =M-x browsel-browse-url= (or set =browse-url-browser-function=
  to it) — route a URL to a browser via the bridge; honors per-URL
  rules in =browsel-url-routes= and focuses an already-open tab
  instead of duplicating it.
- Org-Babel =browsel-js= source blocks that run JavaScript in the
  active tab and return the value into the org buffer.
- An elisp API to query and control the browser: list tabs, focus,
  open, close, evaluate JS with a return value. See the code of
  ~browsel-tab-manager~ to see how to use it.

** Extensibility

- New actions can be added to the extension.  A new action is a single =config.json= edit plus =make build=; no browser
  extension code change is required.

* Keyboard shortcuts

Bind any of the menu actions at =chrome://extensions/shortcuts=.  A
few defaults are pre-suggested:

- =Ctrl+Shift+C= → org-capture
- =Ctrl+Shift+R= → org-roam-capture
- =Ctrl+Shift+W= → open in eww
- =Ctrl+Shift+G= → save ChatGPT conversation

The remaining commands are declared but unbound — Chrome only honors
four default-bound shortcuts per Chrome extension, so the rest must
be assigned manually.

* Per-tab eval consent

The toolbar icon is purple by default.  It turns *red* on tabs that
have granted Emacs permission to evaluate JavaScript on them.  Closing
the tab revokes the grant.

The first =EVAL_IN_ACTIVE_TAB= request on a tab displays an in-page
overlay showing the code about to run, with four buttons: *Deny*,
*Allow once*, *Allow 1 hour*, *Allow this tab*.  30 seconds without
a response counts as deny.

* Querying and controlling the browser from Emacs

The same =browsel-request= function used for JavaScript
evaluation also invokes the other handlers the browser extension
exposes.  The built-in handler names cover the operations Emacs is
most likely to need: =GET_ALL_TABS= returns the current tab list,
=GET_ACTIVE_TAB= returns just the focused tabs of each window,
=OPEN_TAB= creates a new tab at a given URL, =FOCUS_TAB= switches to
an existing tab, and =CLOSE_TAB= closes one.  Each handler's payload
keys and response shape are documented in =CONTRIBUTING.org=.

#+begin_src emacs-lisp
;; List every open tab as title/url pairs
(mapcar (lambda (tab) (cons (plist-get tab :title) (plist-get tab :url)))
        (browsel-request "GET_ALL_TABS" nil))

;; Switch to a tab by id (and focus its window if it is in another window)
(browsel-request "FOCUS_TAB" '(:id 1234 :focusWindow t))

;; Open a new tab
(browsel-request "OPEN_TAB" '(:url "https://example.com"))
#+end_src

** Quick helpers

Three convenience commands wrap the most common
"grab-from-the-active-tab" operations.  Each does double duty:
called interactively it inserts the result at point and returns
nil; called from Lisp it returns the value as a string and inserts
nothing.

| Command                       | Interactive use                              | Lisp return value                       |
|-------------------------------+----------------------------------------------+-----------------------------------------|
| =browsel-org-link=     | inserts =[[URL][TITLE]]= for the active tab at point | the link string                        |
| =browsel-selection=    | inserts the active tab's selected text at point | the selection (empty string when none) |
| =browsel-scroll-page=         | scrolls the active tab one viewport (sign-honored numeric prefix; bare =C-u= = back) | nil (the eval response)                |
| =browsel-video-toggle-play=   | toggles play/pause on the first =<video>= element | interactive only                        |
| =browsel-video-advance=       | seeks the first =<video>= by 5 seconds (numeric prefix N; bare =C-u= = back) | interactive only                        |
| =browsel-video-current-time=  | inserts the first =<video>='s current time (=H:MM:SS= / =M:SS= / =S=) | the timestamp string                    |
| =browsel-youtube-org-link= | inserts =[[URL&t=Ns][TITLE @ TIMESTAMP]]= for the active YouTube tab | the link string                         |

Lisp examples:

#+begin_src emacs-lisp
;; Build an org link for whatever is in the active tab.
(let ((link (browsel-org-link)))
  ...)

;; Read the user's current page selection.
(let ((quote (browsel-selection)))
  ...)

;; Programmatic scroll, three pages down.
(browsel-scroll-page 3)
#+end_src

When two browsers are connected, the helpers (and =browsel-tab-manager=)
use =browsel-default-client= as the default.  Set it once in your
config or change it interactively:

#+begin_src emacs-lisp
;; permanent config
(setq browsel-default-client "chrome")
#+end_src

Or =M-x browsel-set-default-client= to pick the active browser for
the rest of the session (=C-u= clears the setting).  If no default
is set and a multi-client command runs, the first prompt's answer
is remembered for subsequent commands in the session.

Programmatic Lisp callers can still talk to either browser at any
time by passing the client name explicitly:

#+begin_src emacs-lisp
(browsel-org-link "firefox")
(browsel-request "GET_ALL_TABS" nil "chrome")
#+end_src

** Route URLs through browsel

=browsel-url-handler= provides =browsel-browse-url=, a function that
sends URLs from Emacs to a connected browser.  Set
=browse-url-browser-function= to it and every URL Emacs opens (org
links, eww, mu4e, dired, …) is routed to a chosen browser according
to =browsel-url-routes=, optionally into an incognito window.  When
a tab matching the URL is already open, =browsel-browse-url= focuses
that tab rather than opening a new one.

#+begin_src emacs-lisp
;; One-liner to send everything through browsel using browsel-default-client.
(setq browse-url-browser-function #'browsel-browse-url)

;; Per-URL routing: first match wins.
(setq browsel-url-routes
      '((:pattern "github\\.com/"
         :client  "chrome")
        (:pattern "app\\.slack\\.com/"
         :client  "chrome"
         :tab-match "app\\.slack\\.com/")
        (:pattern "news\\.ycombinator\\.com/"
         :client  "firefox"
         :incognito t)))
#+end_src

Each route plist accepts:

| Key          | Required | Purpose                                                          |
|--------------+----------+------------------------------------------------------------------|
| =:pattern=   | yes      | Regex matched anywhere in the URL (no anchors needed)            |
| =:client=    | yes      | Client to address (=chrome= / =firefox=)                        |
| =:incognito= | no       | When =t=, open in an incognito / private window                  |
| =:tab-match= | no       | Override for the already-open tab detection; absent → tab URL must contain the requested URL |

Routing (which client) and tab matching (which open tab) are
separate concerns:

- *Routing* uses =:pattern=.  The first matching route picks the
  client and the incognito flag.  No match → =browsel-default-client=,
  non-incognito.
- *Tab matching* defaults to substring: the tab URL must contain
  the requested URL.  A sub-page of the requested URL still counts
  (opening =https://github.com/x/y= focuses a tab on
  =https://github.com/x/y/issues/3=), but an unrelated page on the
  same domain does not.  Bare-domain requests get a small
  relaxation: the requested host (with leading =www.= stripped) is
  the substring needle, so opening =https://amazon.ca/= still
  reuses an open tab on =https://www.amazon.ca/dp/B1234=.  A route's
  =:tab-match= regex overrides this default when the default isn't
  the policy you want.

When the chosen client has a matching tab, it is focused and the
window raised.  Otherwise a fresh tab is opened — in an incognito
window for incognito routes, in the active window otherwise.

*Incognito caveat.*  Opening an incognito tab requires the user to
toggle *Allow in incognito* for the browsel extension on
=chrome://extensions= (or the Firefox equivalent).  Without that
toggle the extension cannot enumerate or create incognito windows;
=browsel-browse-url= detects the failure, warns, and falls back to a
normal tab so the URL still reaches the user.

** Jump to a tab

=M-x browsel-tab-manager= lists every open tab and
focuses the pick.  Each row shows three columns: a flag prefix
(=a=ctive, =s=ound, =i=ncognito; space when unset), the URL host,
and the page title — each carries its own face so they read as
distinct columns.

#+begin_example
[a  ] mail.google.com                 Inbox (28) - xyz@gmail.com
[ s ] www.youtube.com                 2026 F1 Barcelona GP race analysis
[  i] duckduckgo.com                  Search results
[   ] github.com                      GitHub - dmgerman/browsel
#+end_example

In-prompt action keys:

| Key     | Action                                                                   |
|---------+--------------------------------------------------------------------------|
| =?=     | help buffer with the legend and the bindings below                       |
| =RET=   | focus the highlighted candidate's tab and window, then exit              |
| =M-RET= | show the highlighted tab in its browser window; stay in the prompt, Emacs keeps focus |
| =M-k=   | close the highlighted candidate's tab (see =browsel-tab-manager-confirm-close=) |
| =C-c c= | copy the highlighted candidate's URL to the kill ring, stay in the prompt|
| =C-t=   | cycle the sort order: =mru= → =title= → =domain= → =window=              |

Sort and typed-text are preserved across =M-k= and =C-t= — both
re-fetch =GET_ALL_TABS= and re-enter the prompt with the same
filter, so a chain of =M-k= presses with no typed text walks down
the list cleanly.

=M-x browsel-tab-manager-close-duplicates= sweeps duplicate tabs
in one shot.  Two tabs are duplicates when their URLs match after
stripping any =#fragment=; query parameters (=?id=1=) are kept
distinct.  Pinned tabs are skipped.  The most-recently-accessed
tab in each duplicate group is kept; the rest are closed after a
=yes-or-no-p= confirmation showing the count.

With one browser connected the tab manager targets that one.
With two connected, set =browsel-default-client= in your config —
or change it mid-session with =M-x browsel-set-default-client=
(prefix arg clears it).  The interactive model is one browser at
a time; programmatic callers can still pass an explicit CLIENT to
the helpers to talk to either.

* Run JavaScript in any tab from Emacs

** =browsel-request=

=browsel-request= is the elisp function that sends a request to
the browser extension and returns its response.  It is the function
you call whenever you want Emacs to invoke a handler in the browser
extension — including running JavaScript in a tab.  The call is
synchronous: it blocks until the browser extension replies, or until
=browsel-request-timeout= (10 seconds by default) elapses,
whichever happens first.

Its signature is:

#+begin_src emacs-lisp
(browsel-request NAME &optional PAYLOAD CLIENT)
#+end_src

=NAME= is a string identifying the handler — for example
="EVAL_IN_ACTIVE_TAB"= to evaluate JavaScript, or ="GET_ALL_TABS"= to
list the open tabs.  =PAYLOAD= is an optional plist whose keys depend
on which handler is named; each handler documents the keys it
recognises.

=CLIENT= is the optional name of a connected browser — currently
="chrome"= or ="firefox"=.  When only one browser is connected, omit
=CLIENT= and the request goes to that browser automatically.  When
more than one is connected, supply =CLIENT= to pick which browser
runs the request; the function signals an error if =CLIENT= names a
browser that is not currently connected, and also signals an error
if more than one browser is connected and =CLIENT= is omitted.  Use
=browsel-connected-clients= to inspect the current roster.

The function returns a plist with two fields you should always check.
=:status= is the string ="ok"= when the request succeeded and
="error"= when something went wrong.  On error, the response also
carries =:message= with a human-readable explanation.  On success,
the rest of the response is specific to the handler — for example,
=EVAL_IN_ACTIVE_TAB= populates =:result= with the value produced by
the script.

When the handler is =EVAL_IN_ACTIVE_TAB=, the payload accepts three
keys:

| Key      | Meaning                                                                |
|----------+------------------------------------------------------------------------|
| =:code=  | string — the JavaScript to evaluate                                    |
| =:tabId= | optional integer — target tab (default: the active tab in the current window) |
| =:world= | optional string — ="USER_SCRIPT"= (default) or ="MAIN"=                |

The =:world= key chooses which JavaScript execution context the
snippet runs in.  ="USER_SCRIPT"= (the default) is an isolated
context: it can read and modify the DOM but does not share =window=
with the page's own code; variables and functions defined by the
page are not visible from the snippet.  This default works on both
Chrome and Firefox.

="MAIN"= shares =window= with the page's own code.  Use it when the
goal is to read page-framework state (for example
=window.ytInitialPlayerResponse= on a YouTube watch page) or to call
functions defined by the page.  ="MAIN"= is *Chrome only* — Firefox
MV2 has no equivalent and the request will fail with a clear error.
A block that needs ="MAIN"= is implicitly Chrome-only.

The response =:result= is an array of =InjectionResult= plists, one
per frame the script ran in.  Each entry carries =:documentId=,
=:frameId=, and =:result= — the value of the script's last
expression.

If blocking the editor is not appropriate (long-running scripts,
calls made from inside another WebSocket callback), use the
asynchronous variant:

#+begin_src emacs-lisp
(browsel-request-async NAME PAYLOAD CALLBACK &optional CLIENT)
#+end_src

It returns the request id immediately and invokes =CALLBACK= with the
response plist once the browser extension replies.  Unlike the
synchronous form, it is safe to use from any context.  =CLIENT= has
the same meaning as in =browsel-request=.

** Minimal example

#+begin_src emacs-lisp
(browsel-request "EVAL_IN_ACTIVE_TAB"
                       '(:code "document.querySelector('video').currentTime"))
;; => (:status "ok"
;;     :result ((:documentId "..." :frameId 0 :result 60.34)))
#+end_src

The first request to a fresh tab causes the browser extension to
display the consent overlay; subsequent calls to the same tab go
through silently until consent is revoked or the tab is closed.

For most uses you only care about the value returned by the
JavaScript expression, not the surrounding response envelope.  The
pattern for extracting it from a single-frame result is:

#+begin_src emacs-lisp
(let ((r (browsel-request "EVAL_IN_ACTIVE_TAB"
                                '(:code "document.title"))))
  (plist-get (car (plist-get r :result)) :result))
;; => "Subscriptions - YouTube"
#+end_src

** =browsel-js= source blocks

For one-off JavaScript snippets, writing out a =browsel-request=
call by hand is more typing than necessary.  The =browsel-babel=
module provides =org-babel-execute:browsel-js=, which wraps the function
behind an org-mode source block: the body of the block becomes the
=:code= payload, and the value of the last JavaScript expression is
returned as the block's result.  Babel handles the response
extraction, so the elisp glue is invisible.

The block accepts four optional header arguments:

| Argument    | Values                          | Default          |
|-------------+---------------------------------+------------------|
| =:world=    | ="USER_SCRIPT"= or ="MAIN"=     | ="USER_SCRIPT"=  |
| =:tab-id=   | integer tab id                  | active tab       |
| =:frames=   | ="all"= for per-frame results   | first frame      |
| =:client=   | ="chrome"= or ="firefox"=       | sole client      |

="USER_SCRIPT"= is the portable default: the block runs in an
isolated world that can read and modify the DOM but cannot see
page-defined globals or functions.  Pass =:world "MAIN"= when the
block needs to read the page's own =window= state or call functions
defined by the page; ="MAIN"= is Chrome only and a block that
specifies it will fail on Firefox.

*Read the current video playback time:*

(In the examples below, please remove the space between # and +. I have included them so they
be properly rendered inside the browser)

#+begin_example
# +begin_src browsel-js
document.querySelector('video').currentTime
# +end_src
#+end_example


*Pause playback and report the new state:*

#+begin_example
# +begin_src browsel-js
const v = document.querySelector('video');
v.pause();
({ paused: v.paused, at: v.currentTime })
# +end_src
#+end_example

*Collect several page properties into one object:*

#+begin_example
# +begin_src browsel-js
({ url: location.href,
   title: document.title,
   selection: getSelection().toString(),
   linkCount: document.links.length })
# +end_src
#+end_example

*Run an =async= function and return its resolved value* (the babel
backend awaits the returned promise):

#+begin_example
# +begin_src browsel-js
(async () => {
  const r = await fetch(location.href);
  return { status: r.status, length: (await r.text()).length };
})()
# +end_src
#+end_example

*Run in a specific tab instead of the active one:*

#+begin_example
# +begin_src browsel-js :tab-id 1234567890
document.title
# +end_src
#+end_example

*Return every frame's result (for pages with cross-origin iframes):*

#+begin_example
# +begin_src browsel-js :frames all
location.origin
# +end_src
#+end_example

*Pick a browser when more than one is connected:*

#+begin_example
# +begin_src browsel-js :client firefox
navigator.userAgent
# +end_src
#+end_example

The =:client= header is the name a browser announced via the
=CLIENT_HELLO= request when it connected (currently ="chrome"= or
="firefox"=).  With one browser connected the block runs there
automatically and =:client= may be omitted; with more than one,
omitting =:client= signals an error and lists the connected names.

* Setup

** Requirements

- Emacs with the =websocket= package available.
- Chrome / Chromium or Firefox.

Per-feature.  Each row is needed *only* if you invoke that feature;
the rest of =browsel= keeps working without it.

| Feature                            | Needed                            | What "missing" looks like                                  |
|------------------------------------+-----------------------------------+-------------------------------------------------------------|
| =ORG_ROAM_CAPTURE= action          | =org-roam= package                | Errors when the user picks the org-roam-capture menu item   |
| =SAVE_PAGE= / =CHATGPT=            | =pandoc= on =$PATH=               | Errors at save time; the right-click action fails          |
| =YOUTUBE_TRANSCRIPT=               | =yt-dlp= on =$PATH=               | Errors at fetch time                                        |
| Rich =YOUTUBE= metadata            | YouTube Data API v3 key           | Capture still works, but without title/duration/description |
| Optional Emacs-side modules        | corresponding =(require …)= in the use-package =:config= | Their handlers aren't registered; that name returns "Unknown request" |

If you don't load a module (e.g. don't =(require 'browsel-youtube)=),
none of its external dependencies are needed.

** Emacs

The elisp files are intended for distribution through MELPA (recipe
in the =melpa/= subdirectory of this repository).  Once accepted,
=M-x package-install RET browsel RET= will install the core
=browsel.el= along with =browsel-www=,
=browsel-chatgpt=, =browsel-youtube=, and
=browsel-babel=.  Until then, clone the repository and point
=:load-path= at it.

The browser extension under =extension/= is not yet distributed through
MELPA; it must be loaded into the Web browser separately (see
*Chrome extension* or *Firefox extension* below).  Installing only
the elisp side will start the
WebSocket server but the popup status will remain Disconnected until
the extension is loaded.

One =use-package= form for the core, with one =require= per optional
feature; comment out any you do not want loaded. Change load-path
to the correct location:

#+begin_src emacs-lisp
(use-package websocket :ensure t)

(use-package browsel
  :load-path "~/.emacs.d/modules/browsel"
  :after websocket
  :config
  (require 'browsel-www)         ; SAVE_PAGE        (web archive)
  (require 'browsel-chatgpt)     ; CHATGPT          (save conversation)
  (require 'browsel-youtube)     ; YOUTUBE + YOUTUBE_TRANSCRIPT
  (require 'browsel-babel)       ; org-babel browsel-js blocks

** Chrome extension

The built extension is included in this repository under
=extension/build/chrome/= and is the directory Chrome's "Load
unpacked" expects.  No build step is required to use it as shipped.

In Chrome:

1. Turn on Developer mode (top right of =chrome://extensions=).
2. Click "Load unpacked" and select =extension/build/chrome/=.
3. Click Details → toggle *Allow User Scripts* on (only needed if you
   want =EVAL_IN_ACTIVE_TAB= / =browsel-js= blocks to work).

The popup should turn green within a few seconds.  If not, click
*Reconnect*.

** Firefox extension

The built Firefox extension is included in this repository under
=extension/build/firefox/= and is a Manifest V2 extension.  As with
the Chrome build, the directory is committed already built; no
build step is required to use it as shipped.  MV2 was the
deliberate choice here: Firefox still supports it indefinitely and
provides a genuine persistent background page, which holds the
WebSocket to Emacs without the workarounds MV3 requires.

To load it:

1. Open =about:debugging= in Firefox.
2. Click *This Firefox* in the left sidebar.
3. Click *Load Temporary Add-on…*
4. Select =extension/build/firefox/manifest.json=.

Temporary add-ons unload when Firefox restarts; a signed XPI build
is on the to-do list.  The =browser_specific_settings.gecko.id= is
already set so a signed install would have a stable identity.

** What works the same in both extensions:

Emacs (right-click menu, popup buttons, keyboard commands, content-script relay) and Emacs → browser (=GET_ALL_TABS=,
=OPEN_TAB=, =FOCUS_TAB=, the capture handlers, etc.) routing.  Each browser identifies itself via =CLIENT_HELLO= and Emacs
targets one of several connected browsers through the =CLIENT= argument as described above.

** What differs:

- =EVAL_IN_ACTIVE_TAB= and =browsel-js= source blocks run in the
  content-script isolated world.  The portable ="USER_SCRIPT"=
  default works on both browsers; passing =:world "MAIN"= is Chrome
  only and a block that needs MAIN will surface a clear error on
  Firefox.
- Firefox MV2 does not require a "Allow User Scripts" toggle —
  =browser.tabs.executeScript= runs without separate user opt-in.

** Customising

Most things are reachable from the popup → Options page (or
=chrome://extensions= → Browsel → Extension options):

- *Raise Emacs* — a checkbox per action.  When checked, Emacs is
  brought to the foreground after that action completes.  Stored in
  browser storage and overrides the bundled defaults without
  requiring a rebuild.
- *Menus* / *Handlers* — raw JSON editors for additions or changes
  beyond the bundled defaults.

Larger changes (new menus, new content scripts, etc.) are made in
=extension/config.json=.  =CONTRIBUTING.org= covers the schema.

* Security

Running an unpacked Web browser extension requires the browser's
developer affordance to be enabled: in Chrome, *Developer mode* in
=chrome://extensions=; in Firefox, *Load Temporary Add-on* in
=about:debugging=.  This is a browser-level requirement, not a
=browsel= feature.  Without it the extension does not run, no
WebSocket is opened, and none of the features below are available.

The security posture of the running extension depends on which
features you want to use:

** Sending content from the Web browser to Emacs

The right-click menu actions, the popup buttons, and the keyboard
shortcuts all transfer content from the Web browser to Emacs at your
explicit initiative: you click or invoke the shortcut, the browser
reads the URL, selection, or page HTML, and the data is sent over
the WebSocket on =127.0.0.1:9130= to a registered handler in Emacs.
There is nothing to prompt for in this direction — the user is the
one initiating each transfer.  No additional browser permissions
need to be enabled beyond the developer affordance described above.

** Querying and controlling tabs from Emacs

The handlers =GET_ALL_TABS=, =GET_ACTIVE_TAB=, =OPEN_TAB=, =FOCUS_TAB=,
and =CLOSE_TAB= let elisp running in Emacs read the tab list, switch
between tabs, open new ones, and close existing ones.  These calls
are not prompted: any elisp executing in your Emacs can invoke them.
The capability is roughly equivalent to giving Emacs the ability to
read its own buffer list — appropriate to the trust you already
place in your editor.  No additional browser permissions are required.

** Evaluating JavaScript in tabs

=EVAL_IN_ACTIVE_TAB= and the =browsel-js= source blocks run arbitrary
JavaScript inside a tab, which can read the DOM, cookies,
=localStorage=, and any in-page state.  Two protections cover this
capability:

1. The Chrome-level toggle *Allow User Scripts* (=chrome://extensions=
   → Browsel → Details).  When this is off, =EVAL_IN_ACTIVE_TAB=
   fails with a clear error; every other feature continues to work.
   Leave it off if JavaScript evaluation is not a capability you need.

2. When *Allow User Scripts* is on, =browsel= still requires
   per-tab consent before any script runs.  The first
   =EVAL_IN_ACTIVE_TAB= request to a tab displays an in-page overlay
   showing the code about to run, with four buttons: *Deny*, *Allow
   once*, *Allow 1 hour*, or *Allow this tab*.  The toolbar icon
   turns red on tabs with active consent, and the popup lists every
   consented tab with per-row revocation.

** Pausing the bridge without uninstalling

Turn Developer mode off in =chrome://extensions=.  The extension
stops, the WebSocket connection terminates, and Emacs reports "no
client connected" if it is invoked.

** The WebSocket itself

The Emacs side binds the WebSocket server to =127.0.0.1= explicitly,
not to all interfaces.  The kernel rejects any connection that does
not originate from the loopback interface, so a process on a different
machine cannot connect even when reachable on the same LAN.  The
browser extension is similarly unreachable: extensions do not accept
network connections at all; they only initiate outbound ones.

Local processes are a different matter.  The WebSocket has no
authentication, so any process on the same machine that can connect
to =127.0.0.1:9130= can invoke any handler that Emacs has registered.
The trust boundary is the same as for any other local-only daemon
you run; a process that already has access to your home directory or
your Emacs server can affect your session in equivalent ways.

For the deeper threat model and what is /not/ protected, see
=CONTRIBUTING.org=.

* Troubleshooting

| Symptom                                   | First thing to try                                                            |
|-------------------------------------------+-------------------------------------------------------------------------------|
| Popup shows Disconnected                  | Click Reconnect.  If still red: reload the browser extension card (Chrome: =chrome://extensions=; Firefox: =about:debugging=).  If still red: =M-x browsel-start=. |
| =EVAL_IN_ACTIVE_TAB= errors                | On Chrome: toggle *Allow User Scripts* on in the extension's Details.  On Firefox MV2: only =:world "USER_SCRIPT"= is supported; =:world "MAIN"= is rejected. |
| Consent overlay never appears             | =chrome://=, =chrome-extension://=, and the Web Store refuse eval — Chrome policy. |
| "no client connected" from elisp          | browser extension isn't connected.  Check the popup; reload if needed.         |
| =browsel-start= says "address in use" | Another process is already on 9130.  =lsof -nP -iTCP:9130 -sTCP:LISTEN= to see which. |
| Save-page produces no notification        | Check =*Messages*=; the handler logs success/failure there.                    |
